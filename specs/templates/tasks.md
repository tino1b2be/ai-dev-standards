# Implementation Plan

## Overview

This plan turns `requirements.md` and `design.md` into an ordered list of implementation tasks. Each leaf task is small enough to execute in one sitting, references the requirements it satisfies, and is checked off as it lands. Replace every `REPLACE_ME` and `R?` with project-specific content before starting work.

- [ ] 1. REPLACE_ME top-level task
  - REPLACE_ME short description of the outcome this task produces. Name the files, modules, or artifacts it creates or changes.
  - _Requirements: R?, R?_

  - [ ] 1.1 REPLACE_ME sub-task
    - REPLACE_ME describe the concrete change. Keep it to a few sentences. Reference design.md sections where relevant.
    - _Requirements: R?_

  - [ ] 1.2 REPLACE_ME sub-task
    - REPLACE_ME describe the concrete change.
    - _Requirements: R?_

  - [ ]* 1.3 REPLACE_ME optional sub-task
    - REPLACE_ME optional work that improves the result but is not required for acceptance.
    - _Requirements: R?_

- [ ] 2. REPLACE_ME top-level task
  - REPLACE_ME short description of the outcome.
  - _Requirements: R?_

  - [ ] 2.1 REPLACE_ME sub-task
    - REPLACE_ME describe the concrete change.
    - _Requirements: R?_

- [ ]* 3. REPLACE_ME optional top-level task
  - REPLACE_ME optional workstream (e.g., nice-to-have docs, extra tests, convenience wrappers).
  - _Requirements: R?_

## Notes

- Tasks marked with `*` (e.g., `- [ ]* 1.3`) are optional and can be skipped without affecting acceptance.
- Every leaf task carries a `_Requirements:_` line mapping it back to `requirements.md`.
- Top-level tasks are numbered `1.`, `2.`, ...; sub-tasks are numbered `1.1`, `1.2`, ... and indented by two spaces.
- Order tasks so scaffolding precedes content, content precedes tooling, tooling precedes tests, and tests precede docs.
- Keep each sub-task small enough to complete and verify in one sitting.
